# HLV Architecture

> **H**uman-led intent. **L**LM-executed implementation. **V**alidation-proven behavior.

HLV enforces a hard separation of responsibilities:

- The `Human layer` owns `WHAT`: artifacts, contracts, constraints, and acceptance criteria.
- The `LLM layer` owns `HOW`: code and internal implementation structure.
- The `Validation layer` owns `PROOF`: test specifications, scenarios, traceability, and gates.

Code is a derived artifact. Contracts are also derived artifacts, generated from human artifacts. The source of truth is: human artifacts + confirmed contracts + validation specs.

---

## Operational Cycle

### Milestone-based

```
hlv milestone new "feature"
    │
    ▼
/artifacts -> /generate -> /verify -> /implement (stage 1) -> /validate
                                   /implement (stage 2) -> /validate
                                   ...
                                   -> hlv milestone done
```

Every change is a milestone with its own contracts, stages, and tests. Stages are units of context and execution.

---

## 1. Core Principles

1. The human provides context in free form. The LLM formalizes it into contracts.
2. Contracts are confirmed by the human before implementation begins.
3. Validation specs are generated from contracts - PROOF is born together with WHAT.
4. The LLM generates code from the plan with support for parallel agents.
5. Validation decides production readiness, not reviewer opinion.
6. Any code regeneration is allowed only through behavioral equivalence checks.

Normative terms:

- `MUST`: mandatory requirement; failing it blocks release.
- `SHOULD`: default requirement; deviation is allowed only with an explicit waiver.
- `MAY`: optional practice.

## 1b. Milestone-based Architecture

Two-level model for incremental work:

### Level 1: Global (shared context)
- `human/glossary.yaml` - domain types (shared language across milestones)
- `human/constraints/` - security, performance, compliance
- `validation/gates-policy.yaml` - validation criteria (profile-dependent: `minimal`/`standard`/`full`)
- `llm/src/`, `llm/map.yaml` - accumulated code

### Level 2: Milestones (each one is a mini-project)
Every change is a standalone work unit:
- A context unit (fits into the context window via stages)
- A commit unit (git branch)
- A validation unit (its own gates)

```
human/milestones/
  001-order-create/
    artifacts/              # raw human context
    contracts/              # formal contracts
    test-specs/             # test specifications
    plan.md                 # table of contents: stages, budget
    stage_1.md              # tasks for stage 1
    stage_2.md              # tasks for stage 2
    traceability.yaml       # REQ -> CTR -> TST -> GATE
    open-questions.md       # unresolved questions
```

### `milestones.yaml`

Tracker for the current milestone + history:

```yaml
project: payments
current:
  id: "003-new-payment-method"
  number: 3
  branch: feature/new-payment-method
  stage: 2
  stages:
    - id: 1
      status: validated
    - id: 2
      status: implementing
history:
  - id: "001-order-create"
    status: merged
    merged_at: "2026-03-01"
```

Stage statuses: `pending` -> `verified` -> `implementing` -> `implemented` -> `validating` -> `validated`

### CLI commands for milestones

```bash
hlv milestone new "payment-method"   # create a milestone + branch
hlv milestone status                 # current milestone + stages
hlv milestone list                   # current + history
hlv milestone done                   # merge and move to history
hlv milestone abort                  # abort the current milestone
```

All existing commands (`check`, `status`, `workflow`, `plan`, `trace`, `gates`, `dashboard`) operate on the current milestone from `milestones.yaml`.

| Layer | Owner | Artifacts | Definition of done |
|---|---|---|---|
| Human (`WHAT`) | Human + LLM generation | Artifacts -> Contracts, Glossary, Constraints, Plan | Human confirmed the contracts, open questions closed |
| LLM (`HOW`) | LLM agents | Source code, tests | Build passes, lint passes, pre-check passes |
| Validation (`PROOF`) | LLM generation + CI/CD | Test specs, Scenarios, Traceability, Gate reports | All mandatory gates are green |

## 3. Human Layer

### 3.1 Artifacts (written by the human)

Free form. No structure requirements. The human writes however is convenient.

```
human/milestones/{id}/artifacts/
  research/           # research, API docs, analysis
  infra/              # DB schemas, configs, constraints
  decisions/          # ADRs, rationale
  media/              # screenshots, diagrams
```

### 3.2 Contracts (generated by the LLM, confirmed by the human)

Format: Markdown with embedded YAML blocks for schemas and NFRs.

```
human/milestones/{id}/contracts/
  order.create.md
  order.cancel.md
```

Required contract sections:

| Section | Purpose |
|--------|-----------|
| **Sources** | Links to source artifacts |
| **Intent** | What it does, for whom, context, quotes from artifacts |
| **Input** | JSON Schema (YAML block) |
| **Output** | JSON Schema (YAML block) |
| **Errors** | Table with codes, HTTP statuses, descriptions, links |
| **Invariants** | Natural language + concrete assertions |
| **Examples** | At least 1 happy path + 1 error case in JSON |
| **Edge Cases** | Specific situations with links to decisions |
| **NFR** | Latency, availability, throughput (YAML block) |
| **Security** | Security rules |

Every statement in a contract MUST reference a source artifact.

### 3.3 Glossary

Single domain dictionary. Canonical types, enumerations, forbidden synonyms.

### 3.4 Constraints

Global constraints reused across all contracts: security, performance, compliance.

### 3.5 Plan

#### Stages

The plan consists of `plan.md` (table of contents) + `stage_N.md` (tasks for each stage).

- A stage is a unit of context (contracts + glossary + test specs < ~40K tokens)
- Tasks inside a stage run by dependency graph (topological sort)
- Stages run sequentially (`stage 1` -> `stage 2` -> ...)
- `/implement` works on one stage at a time

### 3.6 Open Questions

Questions the LLM could not infer from artifacts. These block `/verify`.

## 4. Validation Layer (PROOF)

Generated from contracts by `/generate`. Not written manually.

### 4.1 Test Specs

For each contract - a test specification:

```
validation/test-specs/
  order.create.md
  order.cancel.md
```

Contains:
- **Contract Tests**: for every error case and happy path from the contract
- **Property-Based Tests**: for every invariant (>=10,000 generations)
- **Edge Case Tests**: for every edge case from the contract
- **Performance Tests**: from the NFR section
- **Security Tests**: from the Security section
- **Gate Mappings**: mapping each test to a gate

### 4.2 Scenarios

Cross-contract test scenarios:

```
validation/scenarios/
  checkout.happy-path.md
  checkout.partial-failure.md
```

Preconditions -> Steps -> Postconditions -> Acceptance Criteria.

### 4.3 Traceability

Full chain: Artifact -> Requirement -> Contract -> Test -> Gate.

```
human/milestones/{id}/traceability.yaml
```

Machine-verifiable:
- No dangling references
- Every requirement is reachable through a contract and a test
- Every test is mapped to a gate

### 4.4 Gate Policies (static)

Configured once during project initialization:

```
validation/
  gates-policy.yaml           # gate thresholds
  equivalence-policy.yaml     # regeneration rules
  traceability-policy.yaml    # traceability rules
  ir-policy.yaml              # IR versioning
  adversarial-guardrails.yaml # adversarial LLM safety
```

### 4.5 Mandatory Gates

| Gate | Threshold |
|---|---|
| Contract tests | 100% pass on mandatory scenarios |
| Property-based tests | >=10,000 generations with no counterexamples per invariant |
| Integration tests | P0: 100%, P1: >=95% |
| Performance | `p99 <= contract.nfr.latency_p99_ms`, error rate `<= 0.1%` |
| Security | 0 Critical/High vulnerabilities |
| Mutation testing | >=70% mutation score for changed modules |
| Observability | Metrics + trace spans + structured logs for all capabilities |

## 5. LLM Layer

### 5.0 CRITICAL: Code Architecture Philosophy

> **Humans do NOT read generated code. Code is written FOR machines - LLM agents read it, LLM agents modify it, automatic gates validate it.**

This changes everything about code structure:

1. **Comments are for LLM navigation.** The contract is the documentation. Comments in code are navigation markers for the LLM, not explanations for humans. Format: `// @ctx: stock validation for order.create contract`. If something needs explaining, the contract is incomplete - go fix the contract.

2. **Maximum LLM readability. No layers.** Flat module structure. Explicit types everywhere. No clever patterns, no metaprogramming, no hidden behavior. **No layered architecture** - controller/service/repository is a pattern for humans. The LLM writes linearly: input -> validation -> logic -> output -> errors. One file, one flow. An LLM agent with a 200K context window MUST understand any module in isolation.

3. **One contract = one module.** Each contract maps to exactly one directory. All code for `order.create` lives in one directory. No cross-contract imports except domain types.

4. **Domain types are the only shared code.** `domain/types` is the ONLY shared code between features. Everything else is self-contained. Duplication across features is PREFERABLE to coupling. **Duplication is the norm**: copy-paste across features is not refactored until it causes real problems (behavior divergence, forgotten updates after a contract change).

5. **Tests live next to the code.** Tests live in the same file as the code (`#[cfg(test)] mod tests {}` in Rust, equivalents in other languages). Every test is traced to a contract invariant, error case, or NFR via an `@hlv` marker. No just-in-case tests. No test helpers that hide behavior. Test code is as explicit as production code. `hlv check` verifies that every error code, invariant, and constraint rule has an `@hlv <ID>` marker in code. A separate `tests/` directory is only for integration tests (cross-contract scenarios).

6. **File names are arbitrary. `map.yaml` is the navigator.** Files may be named `01.rs`, `handler.rs`, `f3a.rs` - any name is allowed. `llm/map.yaml` is the only source of truth for what files are for. The LLM finds code by descriptions in `map.yaml`, not by file names. Descriptions MUST be sufficient to choose a file without opening it. Each file does one thing, stays under 300 lines, and can be fully replaced by an LLM without understanding neighboring files.

7. **No abstractions for the future.** No base classes, generic frameworks, or plugin systems unless the contract explicitly requires extensibility. Use the simplest code that satisfies the contract. Three similar lines are better than premature abstraction.

8. **Error paths are first-class citizens.** Every error from the contract's Errors table has an explicit code path. No catch-all handlers. No `unwrap()` / `expect()` in production code.

9. **Deterministic PUBLIC API, flexible internal structure.** Given the same contract, two different LLM agents MUST generate code with the same public API (function signatures, error types, inputs/outputs). Internal file structure, file names, and organization are up to the agent. `map.yaml` documents where everything lives.

10. **Machine-verifiable correctness.** Every invariant MUST be testable with property-based tests. Every NFR MUST be measurable. If it cannot be checked automatically, it does not belong in code; it belongs in the contract's open questions.

### 5.1 Entry Point: `project.yaml` + `milestones.yaml`

Every LLM agent MUST start by reading `project.yaml` (global configuration), then `milestones.yaml`.

**`project.yaml`** contains global data:
- **paths** - where directories and files live; `paths.llm.map` points to `llm/map.yaml`
- **stack** - technical stack: components with types, languages, and typed dependencies

**`milestones.yaml`** contains:
- **current** - current milestone: id, branch, stage, stages with their statuses
- **history** - completed milestones

Status, contracts, plan, and questions live inside `human/milestones/{id}/`.

Updated automatically after every `/generate` and `/verify`.

**Language selection policy.** HLV prefers strict, compile-time-safe languages where they naturally fit the task. For backend/service/CLI/system components, the default recommendation is to start with strict, compile-time-safe languages that have a strong ecosystem fit.
This is not dogma: for UI/frontend, bot/automation tasks, glue code, ML/data/AI-chain workloads, and SDK-centric integrations, TypeScript, Python, or another language can and should be chosen if the ecosystem fit is clearly better. Python is not the architectural default, but it can be the best deliberate choice for ML and complex AI chains because of ecosystem maturity. If language choice affects architecture and does not follow from artifacts, the agent must raise it as an open question instead of forcing a language.

### 5.2 Code Traceability (`@hlv` markers)

Every error code, invariant, and constraint rule from contracts MUST have an `@hlv <ID>` marker next to the test that verifies it:

```rust
// @ctx: stock validation for order.create contract
// @hlv OUT_OF_STOCK
#[test]
fn test_out_of_stock() { ... }

// @ctx: transactional write - 3 tables in one tx
// @hlv atomicity
#[test]
fn test_atomicity() { ... }

// @hlv prepared_statements_only
#[test]
fn test_no_sql_injection() { ... }
```

`@ctx` comments are navigation hints for the LLM. They help understand context quickly without reading the entire file. They are optional, but recommended for complex logic.

`hlv check` automatically collects all IDs from `contracts/*.yaml` (`errors[].code`, `invariants[].id`) and `constraints/*.yaml` (`rules[].id`), then scans `src/` and `tests/` for markers. Missing markers produce warning `CTR-010`; in phase `implemented` and later they block `/validate`.

### 5.3 Agent Protocol

1. Read `project.yaml`
2. Find your task (by agent slot or the first free task in the current group)
3. Check that the task's `depends_on` entries are completed
4. Read the contract, test spec, glossary, and dependent code
5. Generate code + tests within the declared output (with `@hlv` markers)
6. Update `project.yaml`: `task.status -> completed`

### 5.4 Coordination Rules

1. The agent generates code + tests within the declared scope
2. Two agents NEVER write to the same file
3. Tasks without dependencies run in parallel
4. At the boundary of a parallel group - create a git commit to hand off artifacts
5. `project.yaml` is updated atomically

## 6. Skills (commands)

| Skill | Purpose | Input | Output |
|-------|-----------|-------|--------|
| `/init` | Scaffold a new HLV project | `--project`, `--owner`, `--agent` | Directories, templates, `project.yaml`, `milestones.yaml`, `.{agent}/skills/` |
| `/artifacts` | Interactive interview -> populate artifacts | artifacts dir | artifacts (tasks, infra, decisions, research) |
| `/generate` | Artifacts -> Contracts + Validation + Plan + Gates coverage | artifacts | contracts, test-specs, `plan.md` + `stage_N.md`, traceability |
| `/questions` | Interactive resolution of open questions with recommendations | `open-questions.md`, artifacts | `open-questions.md` |
| `/verify` | Structure + semantics + coverage + gates<->contracts cross-check | Contracts, test specs, plan, gates-policy | `validation/verify-report.md` |
| `/implement` | Stage -> Code + Tests (parallel agents). Also executes remediation tasks | `stage_N.md`, contracts | `llm/src/`, `llm/tests/` |
| `/validate` | Run gates -> diagnostics -> FIX tasks or release. Two-phase: milestone gates -> global scenarios | `llm/src/`, `llm/tests/`, gate policies | `validation/validate-report.md`, FIX tasks |

All skills work through milestones (`milestones.yaml` with the `current` section).

### Separation of responsibility

Each skill does exactly one thing:

- **`/generate`** - formalizes artifacts into contracts. Checks gate coverage: every gate from `gates-policy.yaml` MUST be covered by contracts or constraints.
- **`/verify`** - validates artifacts and sets the stage status to `verified`. Cross-check: if a mandatory gate has no coverage in contracts/constraints, it is a CRITICAL issue.
- **`/implement`** - executes plan tasks. No diagnostics - only code and tests.
- **`/validate`** - diagnoses problems and plans fixes. No autonomous repair - only FIX tasks for `/implement`.

### Typical cycle

```
/init -> /artifacts -> /generate -> /questions -> /verify -> fix -> /verify -> READY -> /implement -> /validate -> release
```

### Remediation loop (when gates fail)

```
/validate (gates failed)
    -> diagnostics + create FIX tasks in plan.groups
    -> status stays `validating`
    -> /implement (executes only pending FIX tasks)
    -> /validate (reruns failed gates)
    -> repeat until all gates pass -> release
```

In this loop, the human only sees **decisions** (if `/validate` created an open question). Technical details (`pip install`, `cargo audit`, etc.) are hidden - tools are installed automatically.

> Open questions can also be resolved through `hlv dashboard` -> Questions tab (`a` answer, `d` defer).

## 7. Controlled Regeneration

Code regeneration is allowed through a behavioral equivalence protocol:

1. Compare against a fixed Test IR and unchanged seed set
2. Normalize nondeterministic fields (`timestamp`, `uuid`, `trace_id`)
3. Equivalence over response codes, state invariants, and side effects
4. Explicit tolerances: numeric values with tolerance, strings strictly equal

Without passing this protocol, regeneration is blocked.

## 8b. Gates Management

Architecture for CRUD management of gates in `validation/gates-policy.yaml`.

**Data model.** `GatesPolicy` contains `Vec<Gate>`. Each `Gate` is a struct with fields `id`, `type`, `mandatory`, `enabled`, `command`, `cwd`, `pass_criteria`. Serialization/deserialization uses serde.

**Uniqueness.** `add_gate()` checks that `id` does not exist in the current list - duplicates are forbidden. `remove_gate()` looks up by id and returns an error if not found.

**Persistence.** `GatesPolicy::save()` serializes the structure to YAML and writes it to `validation/gates-policy.yaml`. The path is taken from `project.yaml -> paths.validation`. All CRUD operations (`add`/`remove`/`edit`/`enable`/`disable`/`set-cmd`/`clear-cmd`/`set-cwd`/`clear-cwd`) call `save()` after mutation.

**Execution.** `gates run [<id>]` executes the `command` of each enabled gate (or the specified one) via `std::process::Command`. Working directory is `cwd` relative to the project root (or the project root if `cwd` is not set). Result: exit code, stdout, stderr. Exit code `0` = pass.

## 8c. Constraint Management

Architecture for managing constraint files in `human/constraints/`.

**Two formats.** Specialized formats (`SecurityConstraints`, `PerformanceConstraints`) are legacy files with fixed structure. The universal format (`ConstraintFile`) is rule-based and works for arbitrary categories (security, compliance, observability, etc.). `hlv constraints` CRUD commands operate on `ConstraintFile`.

**CRUD.** `hlv constraints add` creates a new YAML file from a `ConstraintFile` template. `add-rule` / `remove-rule` mutate the `rules[]` array inside an existing file. All operations follow: read file -> deserialize -> mutate -> serialize -> write.

**`hlv check` checks.** `CST-010` - file from `project.yaml -> constraints` not found. `CST-020` - duplicate `rule.id` values inside the file. `CST-030` - invalid severity. `CST-010`/`020` are errors (block `/verify`), `CST-030` is a warning.

**Contract linkage.** Contracts refer to constraint rules through `depends_on_constraints`. `hlv check` (`CTR-010`) verifies that every `rule.id` from constraints has an `@hlv` marker in code.

## 8d. Git Integration

Git integration: branch policy, commit conventions, and milestone-level overrides.

**`GitPolicy`.** Struct in `project.yaml` (required field, not `Option`). Fields: `branch_per_milestone` (`bool`), `commit_convention` (`conventional`/`simple`/`custom`), `merge_strategy` (`squash`/`merge`/`rebase`), `branch_prefix` (`string`), `custom_template` (`Option<String>`).

**Branch creation.** On `hlv milestone new "name"` - if `branch_per_milestone: true`, git branch `{branch_prefix}{name}` is created (for example `feature/name`). The branch is stored in `milestones.yaml -> current.branch`.

**`MilestoneGitConfig`.** Optional struct in `milestones.yaml -> current.git`. Lets you override `branch` and `commit_convention` for a specific milestone. If unset, values from `GitPolicy` are used.

**Commit messages.** `hlv commit-msg` reads the convention (from milestone override or global policy), determines the current milestone and stage, and generates a message. Formats: `conventional` -> `feat(001-order): stage 1 - description`, `simple` -> `[001-order] description`, `custom` -> from `custom_template`.

## 8e. MCP Server

Programmatic access to HLV data and operations through the Model Context Protocol.

**Architecture.** The `hlv mcp` subcommand launches an MCP server on top of the existing codebase. No changes to the existing code are required - the MCP layer is just a set of wrappers.

```
┌─────────────────────────────────────────────────┐
│  MCP Clients                                    │
│  ┌──────────┐  ┌──────────┐  ┌──────────────┐  │
│  │Claude Code│  │ Web UI   │  │ Custom tools │  │
│  └────┬─────┘  └────┬─────┘  └──────┬───────┘  │
│       │stdio         │SSE            │stdio/SSE │
└───────┼──────────────┼───────────────┼──────────┘
        │              │               │
┌───────▼──────────────▼───────────────▼──────────┐
│  HLV MCP Server (src/mcp/)                      │
│  ┌────────────┐  ┌──────────┐  ┌────────────┐   │
│  │ Resources  │  │  Tools   │  │  Watcher   │   │
│  │ (read-only)│  │ (ops)    │  │ (notify)   │   │
│  └─────┬──────┘  └────┬─────┘  └──────┬─────┘   │
```

**Resources** - 12 static + 6 parameterized URIs for reading project data.

## 9. Integration with Handoff

HLV integrates with [Handoff](../../) to orchestrate parallel agents:

| Handoff MCP Tool | When it is called | What it does |
|------------------|-----------------|------------|
| `handoff_register` | `/implement` - agent startup | Registers the agent, workspace, and task |
| `handoff_check` | Before writing a file | Checks for conflicts with other agents |
| `handoff_done` | Task completion | Signals completion and triggers change propagation |
| `handoff_status` | Monitoring | Status of all agents and tasks |

Handoff solves the "blind agents" problem: Agent A changes shared code -> Agent B is automatically notified and adapts its work.

## Key Idea

HLV does not force the human to formalize. The human provides context in whatever form is convenient.
The LLM formalizes it, the human confirms it. Validation is born together with contracts.
Code is a third-order derived artifact: `artifacts -> contracts -> code`.
Code is written for LLMs, not for humans. File names are arbitrary, `map.yaml` is the navigator, tests live next to code, and there are no layers.

Three "compilations":
1. **Artifacts -> Contracts + Validation** (`/generate`) - the LLM formalizes intent
2. **Contracts -> Code** (`/implement`) - the LLM implements contracts
3. **Code x Validation -> Release** (`/validate`) - CI/CD proves correctness

If `/validate` finds problems, it creates FIX tasks in the plan and sends them back to `/implement`. The cycle repeats until all gates pass. In that loop, the human only makes decisions and does not perform technical actions.
